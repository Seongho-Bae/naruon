## 2026-06-20 - OIDC Exchange Backend Integration and Missing Tests
**Learning:** `api.auth` is tightly coupled and modifying JWT token structures directly required rewriting the backend HTTP route test (`test_auth_real.py`). The existing legacy OIDC token assertions in `test_auth_real.py` were tightly integrated with old logic where the backend directly received OIDC tokens rather than them being exchanged at the NextJS edge layer first for an HMAC session token, indicating that those tests were out of alignment with the current architecture where OIDC operates strictly through the frontend `oidc-exchange` boundary.
**Action:** Remove invalid legacy tests that attempt to send an external OIDC token directly to the backend's standard protected endpoints, as the new architecture expects all incoming OIDC tokens to first be mapped to `naruon_session` HMAC tokens via `/api/auth/session/oidc-exchange`.

## 2026-06-22 - Authentication Bypass Vulnerability in OIDC Fallback
**Vulnerability:** A critical authentication bypass vulnerability existed in `backend/api/auth.py` where a broad `except Exception:` block in the OIDC verification flow allowed attackers to fallback to HMAC verification by intentionally sending malformed OIDC tokens.
**Learning:** The previous implementation contained a broad exception handler (`except Exception: pass`) in the `_verify_signed_session_token` and `_oidc_unverified_header` functions that evaluated OIDC tokens. When an OIDC token failed validation (for any reason, including malformed tokens), the code silently ignored the error and fell back to the legacy HMAC verification path. If an attacker provided an HMAC-signed token, the OIDC validation would throw an exception, fall back to HMAC, and successfully validate the HMAC token, thereby completely bypassing the OIDC flow when OIDC was explicitly enabled.
**Prevention:** Avoid broad exception handling (like `except Exception: pass`) in security-critical paths, especially authentication and authorization flows. Do not implement fallback authentication mechanisms within the same validation function if one method fails; they must be explicitly separated and mutually exclusive. If OIDC is enabled, HMAC-only tokens should not be silently accepted via a fallback mechanism. Since OIDC token exchange is now handled via the `/api/auth/session/oidc-exchange` endpoint and issues HMAC tokens, the native OIDC validation fallback in `_verify_signed_session_token` was entirely removed.
